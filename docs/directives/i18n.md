# i18n Directive — bruna-is

This document defines the rules for bilingual content, translation, and locale handling on the bruna-is site. Treat it as the single source of truth. Deviations require a documented rationale where the deviation lives.

The site is bilingual: **Icelandic (`is`) is the primary language** and serves at the site root; **English (`en`)** serves under `/en/`. Both versions ship from day one.

---

## 1. URL strategy

Icelandic at the root, English under `/en/`:

```
/                    → IS home
/about/              → IS about
/thjonusta/foo/      → IS service detail
/en/                 → EN home
/en/about/           → EN about
/en/thjonusta/foo/   → EN service detail
```

**Slugs use Icelandic terms in both locales.** `/thjonusta/` (services), `/geirar/` (sectors), `/greinar/` (articles), `/verdreiknir/` (quote calculator) — these are the same on the EN side. This is a deliberate choice: a single set of public URLs keeps inbound links, social shares, and analytics consistent across locales. Visual translation of labels happens at the template layer via the `i18n` filter; URLs do not translate.

Slug stems within each collection are identical across locales — `fireproofing.md` exists in both `src/content/is/thjonusta/` and `src/content/en/thjonusta/` so `locale_links` pairs them automatically.

## 2. Plugin stack — what does what

Three plugins, in registration order in `eleventy.config.js`:

| Plugin | Role |
|---|---|
| `@11ty/eleventy` → `I18nPlugin` | Built-in. Adds `page.lang`, the `locale_url` filter (rewrites links to the current locale), and the `locale_links` filter (lists alternates for the current page). Manages **links between localized pages**; does not translate strings. |
| `eleventy-plugin-i18n` | Loads the UI string dictionary from `src/_data/i18n.js`. Registers the `i18n` filter. **Its filter is overridden — see §3.** |
| Inline `i18nOverride` | Replaces the upstream `i18n` filter with a corrected implementation. Defined as a tiny plugin so registration order guarantees it runs *after* the upstream filter is added. |

Config (`eleventy.config.js`):

```js
eleventyConfig.addPlugin(I18nPlugin, {
  defaultLanguage: "is",
  errorMode: "allow-fallback",
});

eleventyConfig.addPlugin(i18nPlugin, {
  translations,
  fallbackLocales: { en: "is" },
});

eleventyConfig.addPlugin(function i18nOverride(ec) {
  ec.addFilter("i18n", function (key, ...rest) { /* see §3 */ });
});
```

`defaultLanguage: "is"` is required: combined with the root-permalink IS strategy, it makes `locale_url` resolve correctly in both directions.

`errorMode: "allow-fallback"` is required: strict mode would fail the build whenever a page exists in only one language.

## 3. The custom `i18n` filter override

The upstream `eleventy-plugin-i18n` filter is broken for this project on two axes. The override (registered as the inline `i18nOverride` plugin in `eleventy.config.js`) fixes both:

1. **Dotted key lookup.** The upstream filter does `` lodash.get(translations, `[${key}][${locale}]`) ``, interpreting dots inside a key as path separators. Every key in our dictionary is dotted (`"ui.skip_to_content"`, `"footer.copyright"`, etc.) so the lookup always misses and returns the raw key. The override uses direct bracket-property access against the same `translations` object.

2. **URL-prefix locale autodetection.** The upstream filter detects locale via `url.split('/')[1]`, which returns `'about'` (not `'is'`) for an IS-at-root page like `/about/`. The override requires the caller to pass `lang` explicitly as the `localeOverride` argument.

**Do not modify or remove the override** without first reading the upstream `eleventy-plugin-i18n` source. If the upstream plugin ships a fix for either issue, you can drop the override — but verify behavior on at least one dotted key and one IS-at-root page first.

## 4. Filter callsite contract

Every template callsite passes `lang` explicitly:

```njk
{# ✅ Correct — `lang` passed as the explicit override #}
{{ "ui.skip_to_content" | i18n(lang) }}
{{ "articles.read_time_minutes" | i18n(lang) }}

{# ❌ Wrong — relies on broken autodetect, will return the raw key #}
{{ "ui.skip_to_content" | i18n }}
```

The signature mirrors the upstream filter for parity:

```
{{ "key" | i18n(lang) }}                    → simple lookup
{{ "key" | i18n({ name: ... }, lang) }}     → with interpolation data
```

The override interpolates `{{name}}` tokens inside translated values when a data object is passed (no callsite uses this today, but the surface is preserved).

When a translated string contains HTML (e.g. `"brand.tagline"` uses `<br>`), append `| safe`:

```njk
{{ "brand.tagline" | i18n(lang) | safe }}
```

Otherwise the HTML escapes and breaks the markup.

## 5. One-way fallback: EN → IS, never IS → EN

Missing EN strings fall back to IS. Missing IS strings render the raw dictionary key (e.g. `ui.skip_to_content`) and warn red — Icelandic is the source of truth, so a missing IS value must surface loudly.

If a raw key appears in rendered HTML, add the entry to `src/_data/i18n.js`. Do not patch around the miss by hard-coding a string in a template.

## 6. Translation miss logging

Every miss writes a line to `_site/.translation-misses.log` via `recordMiss(key, lang)` and prints a red `console.warn`:

```
[i18n] Translation for 'foo.bar' in 'en' not found.
[i18n] Missing 'en' for 'baz.qux'. Using 'is' fallback.
```

The post-build check (`scripts/check-build.js` step 9) reads this sidecar file and prints `[check-build] WARN: N translation fallbacks (see <path>)`. Fallbacks are warnings, not errors — investigate before merging, but don't block CI on them.

A hard miss (IS missing on an IS page) is also only a warning today. Treat it as a soft error: review the log before every release.

## 7. Dictionary structure

`src/_data/i18n.js` exports a flat object keyed by dotted name:

```js
export default {
  "ui.skip_to_content": { is: "Hoppa yfir á efni",  en: "Skip to content" },
  "nav.services":       { is: "Þjónusta",           en: "Services" },
  "footer.copyright":   { is: "Öll réttindi áskilin.", en: "All rights reserved." },
  // …
};
```

Key conventions:

- **Dotted namespacing**: `ui.*`, `nav.*`, `footer.*`, `cta.*`, `quote.*`, `articles.*`, `utility.*`, `brand.*`, `lang.*`. Use the existing namespace when extending; add a new one only when a UI section grows ≥ 3 keys.
- **Snake_case inside the namespace**: `skip_to_content`, not `skipToContent` or `skip-to-content`.
- **Both locales always populated.** Even if both languages happen to use the same word ("EN", "IS"), write the value twice — never omit a locale.

When adding a new key, place it next to siblings in its namespace, not at the end of the file. The file is hand-curated, not generated.

## 8. UI strings versus page content

This is the boundary between when to use the `i18n` filter and when to put content in page frontmatter:

| Content type | Mechanism |
|---|---|
| Chrome reused across many pages (nav labels, footer headings, "Read more", "Skip to content", "Back to top") | `i18n` dictionary key |
| CTA button labels reused across pages | `i18n` dictionary key (`cta.get_quote`, `cta.contact`) |
| Form labels and validation strings | `i18n` dictionary key |
| Page-specific copy (hero heading, body prose, service descriptions, article content) | Page frontmatter (`heroHeading`, `statement`, `summary`, `cta.heading`) — written once per locale in the per-locale source file |

**Never** hard-code a translatable string in a template literal. If a string is user-visible text and changes between locales, it goes either in the dictionary (chrome) or in frontmatter (page copy). Asset paths, class names, technical attributes don't count as translatable.

## 9. Locale parity rule

Every IS page **must** have an EN sibling at the mirrored path. The source trees under `src/content/is/` and `src/content/en/` are structurally identical: matching directories, matching filenames, parallel frontmatter shape.

```
src/content/is/thjonusta/fireproofing.md
src/content/en/thjonusta/fireproofing.md   ← required
```

If a piece of content genuinely exists only in IS:

- Create the EN sibling anyway with translated frontmatter and body. The build's parallel-slug check (`check-build.js` step 7) prints a WARN line for every unpaired slug. Don't ship orphaned pairs.
- The language switcher uses `locale_links`, which only lists the alternate when it exists. An orphaned page works — but the user is sent to a stub on the alternate side. The fix is to write the sibling.

**Exempt from parity:** pages with `permalink: false` (the about-page subcollections — `milestones/`, `principles/`, `team/`). They emit no HTML, so there's nothing for `check-build.js` to compare; the IS-only data files are intentional and do not trigger the parallel-slug warning.

**Adding an IS page requires adding the EN sibling in the same commit.** This is enforced socially, not by tooling.

## 10. `lang` propagation — the `requireLang` guard

`lang` is set per locale by `src/content/{is,en}/{is,en}.11tydata.js` and cascades to every descendant. `base.njk` renders `<html lang="{{ lang | requireLang }}">` — the filter throws if `lang` is missing so a misplaced page fails the build loudly rather than rendering `undefined` into meta tags and JSON-LD.

Rationale lives in the inline comment at `eleventy.config.js` `requireLang` (`addFilter("requireLang", …)`). Do not bypass the filter; give a one-off page an explicit `lang: "is"` in its own frontmatter if it lives outside the locale tree.

## 11. `<html lang>`, `hreflang`, and `x-default`

Every page in `base.njk` emits:

```njk
<html lang="{{ lang | requireLang }}">
<link rel="alternate" hreflang="{{ lang }}" href="{{ meta.url }}{{ page.url }}">
{%- for link in page.url | locale_links %}
<link rel="alternate" hreflang="{{ link.lang }}" href="{{ meta.url }}{{ link.url }}">
{%- endfor %}
{%- if lang == "is" %}
<link rel="alternate" hreflang="x-default" href="{{ meta.url }}{{ page.url }}">
{%- else %}
{%- for link in page.url | locale_links %}{% if link.lang == "is" %}
<link rel="alternate" hreflang="x-default" href="{{ meta.url }}{{ link.url }}">
{%- endif %}{% endfor %}
{%- endif %}
```

The `x-default` alternate always points to the IS version: search engines treat IS as the canonical fallback when a user's language preference matches neither IS nor EN.

`locale_links` only returns languages where the alternate exists. Trust it — do not add manual fallback logic in the template.

## 12. Per-locale metadata

Site metadata is keyed by locale via `src/_data/meta.js`:

```js
const byLocale = {
  is: { title: "...", description: "...", ogLocale: "is_IS", addressCountry: "Ísland" },
  en: { title: "...", description: "...", ogLocale: "en_US", addressCountry: "Iceland" },
};
```

Templates access via `meta.byLocale[lang]`. Shared values (URLs, contact phone/email, deployment coordinates) live at the top level of the export, not in either locale.

When adding a new metadata field that varies per locale, add it to **both** `byLocale.is` and `byLocale.en`. Don't add a key to only one and let the other fall through to `undefined`.

## 13. JSON-LD per page

Every JSON-LD block emitted by a layout includes `"inLanguage": "{{ lang }}"`:

```njk
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "{{ title | jsonEscape }}",
  ...
  "inLanguage": "{{ lang }}"
}
</script>
```

All user-controlled strings going into JSON-LD pass through `jsonEscape`. See [`templates-and-layouts.md`](./templates-and-layouts.md) §7.

## 14. Date formatting

The `dateDisplay` filter takes `lang` as an argument:

```njk
<time datetime="{{ date | dateIso }}">{{ date | dateDisplay(lang) }}</time>
```

Produces:
- `lang == "is"` → `15. janúar 2026` (Icelandic locale)
- `lang == "en"` → `January 15, 2026`

`dateIso` is locale-neutral: `2026-04-02`. Use it for `datetime` attributes and JSON-LD `datePublished` / `dateModified`.

## Enforcement appendix

`scripts/check-build.js` (run by `npm run build`) machine-checks the following:

- **§9 (locale parity, soft):** every IS slug has an EN parallel and vice versa; mismatches print `[check-build] WARN`.
- **§10 (`requireLang`):** every HTML output page has `lang` set, or the build throws via the filter.
- **§6 (translation misses):** post-build, prints `[check-build] WARN: N translation fallbacks` if the sidecar log has entries.

Conventions enforced by review (no automated check):

- §1 URL strategy: IS slugs reused in EN.
- §4 callsite contract: every `i18n` filter call passes `lang` explicitly.
- §3 override: do not modify without reading upstream source.
- §5 fallback direction: never add an IS → EN fallback.
- §7 dictionary structure: both locales populated for every key.
- §8 boundary: UI strings via dictionary, page copy via frontmatter.
