# CSS consistency review — Eleventy port of Brunaþéttingar mockup

**Reviewer:** consistency-auditor
**Date:** 2026-05-18
**Scope:** CSS layer (`src/assets/css/*`, font assets, base layout, and content templates that reference CSS)
**Authoritative directives:** `docs/instructions/FRAMEWORK-PORT-PROMPT.md`, `docs/instructions/FRAMEWORK-I18N.md`, plan at `/Users/olafur/.claude/plans/read-these-three-files-eager-feather.md`
**User focus:** CSS correctness; the user reports the typeface looks different between IS and EN versions of the live site

---

## TL;DR — the typeface symptom

**Inter is not loaded on either locale.** It is fetched as a preload but never registered as a face, so every page falls back to the system stack (`-apple-system, BlinkMacSystemFont, "Segoe UI", system-ui, sans-serif`).

Both fonts files at `src/assets/fonts/inter-{400,600}.woff2` are the **full** Inter 4.001 from rsms, not the Latin-only docs subset that the dispatch brief suspected. Their cmap contains every Icelandic glyph (þ, ð, æ, Þ, Ð, Æ, ö, á, é, í, ó, ú, ý and their capitals — verified). The brief's hypothesis #1 (Latin-only subset) is **false**: the fonts would have rendered IS correctly if they had loaded at all.

The actual mechanism is in Finding 1 below: the `@font-face` `src:` URL in `tokens.css` is host-absolute (`/assets/fonts/inter-400.woff2`) and the site is deployed under the `/bruna-is/` GitHub Pages project path. The browser requests `https://brunathettingar.github.io/assets/fonts/inter-400.woff2` and gets a 404. Inter never registers.

Why the user perceives a *difference between* locales rather than "the font looks wrong on both": with Inter falling back, the browser per-glyph substitution chain runs against the system stack. The EN page is all ASCII; the IS page has Icelandic codepoints in headings (`Brunaþéttingar`, `tæknieinangrun`, `Þjónusta`, etc.) and the system fallback chain can pick a different physical font for those specific glyphs than for the surrounding ASCII — producing visible mixed-font rendering on IS pages that doesn't occur on EN pages.

Fix priority: this is one CSS line, ships immediately. See Finding 1.

## Findings

### Finding 1 — `@font-face` `src:` URL ignores the project pathPrefix (the typeface mismatch root cause)
**Severity:** Critical
**Files:** `src/assets/css/tokens.css:3, 10`
**Directive:** FRAMEWORK-PORT-PROMPT.md §"Conventions" — "Web fonts only if the mockup uses them. If it does, self-host via `assets/`… and add `<link rel="preload">` for the primary face." Plan §5 — "self-host Inter 400 + 600 under `src/assets/fonts/`, `font-display: swap`, preload 400 in `base.njk`."

The site ships from a GitHub Pages project subpath. `eleventy.config.js:169` sets `pathPrefix: "/bruna-is/"` and adds `HtmlBasePlugin` (line 13) to rewrite HTML `href`/`src` accordingly. `HtmlBasePlugin` only rewrites attributes inside HTML output — it does not touch `url(...)` references inside external stylesheets.

`src/assets/css/tokens.css` declares:

```css
@font-face {
  font-family: "Inter";
  src: url("/assets/fonts/inter-400.woff2") format("woff2");
  ...
}
```

At runtime the browser resolves this to `https://brunathettingar.github.io/assets/fonts/inter-400.woff2` (host-absolute, no `/bruna-is/`). That path 404s — the real file is at `/bruna-is/assets/fonts/inter-400.woff2`. Inter is never installed in the page's font registry.

The `<link rel="preload" as="font" type="font/woff2" href="/bruna-is/assets/fonts/inter-400.woff2" crossorigin>` in `base.njk:18` *does* get prefixed (because HtmlBasePlugin processes `href`), so the preload succeeds — but the preload is orphaned. Preload matches consumers by exact URL; the `@font-face` requests a different URL and never sees the preloaded bytes.

This is the **root cause of the user's reported typeface symptom**. With Inter missing:
- All text falls back to `-apple-system, BlinkMacSystemFont, "Segoe UI", system-ui, sans-serif` (`tokens.css:44` and `main.css:23`)
- On EN pages the body text is ASCII-only, so a single system font renders the whole line
- On IS pages, headings like `Brunaþéttingar & tæknieinangrun.` mix ASCII and Icelandic codepoints. The browser's per-glyph substitution can pick a different physical font for `þ`, `ð`, `æ` etc. than for the surrounding ASCII glyphs, especially on platforms where the first stack entry doesn't cover Icelandic in the same weight. The result is the visible "different typeface" the user reports.

Fix (one of):
- Change `src/assets/css/tokens.css` to `src: url("../fonts/inter-400.woff2")` and `inter-600.woff2` — relative from the stylesheet location works regardless of pathPrefix.
- Or, add a font-specific transform on the built `tokens.css` that prefixes `url(...)` inside `@font-face`.
- Or, move the `@font-face` rules into an inline `<style>` block in `base.njk` so HtmlBasePlugin or template interpolation handles the prefix (this collides with the framework's "no inline `<style>`" rule, so not recommended).

The relative-URL form is the cleanest fix and is what self-hosted-under-`assets/` implies.

After the fix, also remove `-apple-system, BlinkMacSystemFont, "Segoe UI", system-ui` from in front of Inter or accept that during FOIT/FOUT the fallback still renders briefly — but at least the final rendering will be Inter on both locales.

### Finding 2 — Spec rule violation: `<picture>` pipeline is completely bypassed
**Severity:** High
**Files:** `src/content/**/index.njk`, `src/content/**/article.njk`, `src/_includes/partials/*.njk`
**Directive:** FRAMEWORK-PORT-PROMPT.md §"Conventions" — "No `<picture>` written by hand. Use `<img>` and let the plugin transform." §"Images" — "Write plain `<img src=... alt=... width=... height=...>` — the `eleventy-img` plugin transforms it..."

Zero `<img>` tags exist anywhere under `src/content/` or `src/_includes/`. Every hero, every service block, every sector card, every article card, every page-hero, and the leading-stats portrait all use `<div class="bg" style="background-image: url('...');"></div>` or `<div class="pic" style="background-image: url('...');"></div>` — 34 occurrences across the content tree.

`@11ty/eleventy-img`'s transform pipeline is configured in `eleventy.config.js:32-39` (AVIF/WebP/JPEG at 400/800/1200/auto widths) but has zero `<img>` to operate on. The dependency is dead weight; visitors get the original ~30+ full-size JPEGs from `src/img/` instead of optimized responsive variants.

This is the mockup's authored style (the mockup also uses background-images), but the framework spec explicitly requires plain `<img>` for the pipeline to transform. The plan's §9 verification list includes "`<img>` rendered as `<picture>` by `eleventy-img`" — that verification step would fail on every page.

Related: FRAMEWORK-PORT-PROMPT.md §"Conventions" says **"No inline styles or `<style>` blocks in templates."** Every `background-image: url(...)` is an inline style. The `prefixInlineUrls` transform in `eleventy.config.js:19-30` only exists because inline styles were used — it's a workaround that papers over a rule violation.

Fix: convert hero/card images to `<img loading="lazy" src="..." alt="..." width="..." height="...">` and let the plugin emit `<picture>`. Use CSS object-fit/position to retain the cropping that `background-size: cover` provides today. Headers with text overlays will need a `position: relative` + absolutely-positioned `<img>` pattern (or `object-fit: cover` with `inset: 0`).

### Finding 3 — Token-only rule: `main.css` carries raw values throughout
**Severity:** High (acknowledged trade-off, but the scope is larger than was implied)
**Files:** `src/assets/css/main.css` (1551 lines)
**Directive:** FRAMEWORK-PORT-PROMPT.md §"tokens.css" — "Downstream CSS **must** reference tokens — no raw hex codes, no magic px values, outside this file." Plan §7 — "Decisions (confirmed)" implies token migration is a follow-up.

Quantitative scope in `main.css`:
- ~52 raw hex codes (most are repeats of the same brand colors that already exist as tokens — `#fff`, `#0a0d12`, `#ee7c1d`, `#1453a8`, `#f1f3f6`, etc.)
- ~37 raw `rgba(...)` calls (white-on-dark overlays at `0.4`/`0.5`/`0.55`/`0.7`/`0.85`/`0.95` etc., scrim gradients — almost all duplicates of tokens already added as `--alpha-on-dark-*` and `--alpha-scrim-*` in `tokens.css:35-42`)
- ~453 instances of raw `px` values (sizes, paddings, margins, gaps, border widths)

This is the mockup CSS imported verbatim. The plan's §5 token inventory explicitly enumerated alpha overlays and `--space-1` through `--space-28` to cover exactly the cases that `main.css` still hard-codes. The plan's roadmap §6 says: "Move mockup styles into per-section files per §5, **swap raw values for tokens**." That swap step has not happened in `main.css`.

Surfacing this with full scope because the dispatch brief noted the plan §7 acknowledged token migration as a follow-up — but the scale (>500 raw values) is large enough that a future maintainer needs to know what's deferred, not just that "some" deferral exists.

Recommend: a separate "token-migration" pass over `main.css` with a focused PR.

### Finding 4 — `:root` redefined in `main.css`, with a parallel and overlapping token naming convention
**Severity:** High
**Files:** `src/assets/css/main.css:6-24` (redefines `:root` after `tokens.css` did)
**Directive:** Plan §5 — "Existing tokens to keep verbatim" lists `--bg`, `--accent`, `--brand-blue`, etc. with the canonical names. Tokens belong in `tokens.css`. FRAMEWORK-PORT-PROMPT.md §"CSS structure" — load order is "tokens.css -> reset.css -> layout.css -> nav.css -> main.css -> per-section". Tokens, by definition, are defined once at the top of the cascade.

Both files declare `:root` with overlapping properties:

`tokens.css:16-128` defines the canonical typography tokens: `--font-body`, `--font-serif`, plus all the `--text-*`, `--space-*`, etc.

`main.css:6-24` then redeclares `:root` with:
- Color tokens that duplicate `tokens.css` (`--bg`, `--bg-soft`, `--bg-dark`, `--text`, `--muted`, `--accent`, `--brand-blue`, etc.) — identical values; the override is silent and accidental.
- **Different** typography token names: `--sans` and `--serif` (vs. `tokens.css`'s `--font-body` and `--font-serif`). Both reference the same physical fonts but live under different identifiers.

Consequences:
1. `reset.css:7` uses `var(--font-body)` (correctly from `tokens.css`). `main.css:29` then sets `body { font-family: var(--sans); }`. Both produce the same render today, but they're orthogonal definitions — changing `--font-body` in `tokens.css` won't affect anything `main.css` styles, and vice versa.
2. ~17 rules in `main.css` use `var(--sans)` or `var(--serif)`. If the design later swaps Inter for another typeface, the change must be made in two places.
3. The color overrides in `main.css:6-24` mean `tokens.css` color values are dead until `main.css` is removed or the redeclaration is dropped.

Pattern drift relative to the spec's intent. `tokens.css` was supposed to be **the** source of truth — that's the whole point of the design-tokens approach. The current state has two sources of truth in cascade order, where the second silently wins.

Fix: delete the `:root` block at `main.css:6-24` and rename `var(--sans)`/`var(--serif)` throughout `main.css` to `var(--font-body)`/`var(--font-serif)` (or add aliases in `tokens.css` and migrate later).

### Finding 5 — Plan §7 confirmed decision NOT implemented: single `services`/`sectors` collection with `featured: bool`
**Severity:** High
**Files:** `src/_data/services.js`, `src/_data/sectors.js`, `src/content/is/index.njk`, `src/content/en/index.njk`, `eleventy.config.js:152-164`
**Directive:** Plan §7 — "Featured model: single `services` and single `sectors` collection with `featured: bool` (no second collection)." Plan §4 — "For each of `services`, `sectors`, ...: directory data file, listing page, layout template, and one `.md` per entry. Custom collections in `eleventy.config.js` for `featuredServicesIs`/`En` and `featuredSectorsIs`/`En`." FRAMEWORK-I18N.md §"Content authoring contract" — paired `.md` files with parallel frontmatter.

Implementation diverges in two compounding ways:

1. Services and sectors live as plain JS data arrays in `src/_data/services.js` and `src/_data/sectors.js` with bilingual fields per entry (`title: { is, en }`, `lead: { is, en }`, etc.). There are no per-entry `.md` files in `src/content/{is,en}/{thjonusta,geirar}/` — only the `index.njk` listing. The FRAMEWORK-I18N.md authoring contract ("same filename = same slug = locale_links pairs them automatically") is bypassed entirely; `locale_links` cannot pair entries that aren't pages.
2. Neither data file has a `featured: bool` field. The home page's four-pillar grid (`src/content/is/index.njk:65-133` and the EN equivalent) is **hardcoded markup** — four pillars statically written, not derived from `services` filtered by `featured`. The plan's mockup inventory §1 explicitly required "featured `services` (4 of 7), featured `sectors` (4 of 8)."
3. `eleventy.config.js:151-165` registers `featuredServicesIs`, `featuredServicesEn`, `featuredSectorsIs`, `featuredSectorsEn` collections that filter by tags `services-is`, `services-en`, etc. Nothing has those tags (no `.md` files in the collection dirs to receive directory-data tags). The four collections are **dead code** — verified by grepping the source tree: zero usages of `collections.featuredServices*` or `collections.featuredSectors*` exist in any template.

Articles follow the same pattern: `src/_data/articles.js` has the `featured` field (used by `greinar/index.njk`), but article detail URLs are generated via Eleventy `pagination` over the data file rather than via a markdown-per-article collection.

This is a deliberate-looking architectural deviation from both the plan and the i18n spec's authoring contract. It works at the output level (pages render), but it means:
- Future editors can't add a service/sector by dropping a `.md` file — they have to edit a JS module.
- The hardcoded home pillars can drift from the services data (changing service #1's title in `services.js` won't update the home page).
- Translation handoff is harder: a translator gets a `.js` file with nested `{ is, en }` objects rather than `.md` files with frontmatter.

Recommend: surface this with the user and decide whether to (a) migrate to markdown collections per spec, or (b) update the plan/spec to ratify the data-file approach. Right now the plan and spec promise one thing, the implementation does another.

### Finding 7 — `responsive.css` uses `!important` 9 times
**Severity:** Medium
**Files:** `src/assets/css/responsive.css:41, 42, 65, 66, 67, 88, 97, 104, 105`
**Directive:** Dispatch brief lists "no `!important`" among the rules. (The framework spec itself does not enumerate this rule — but the brief treats it as part of the rule set.)

Examples:

```css
.statement .row, ... {
  grid-template-columns: 1fr !important;
  gap: var(--space-8) !important;
}

.statement h2, ... {
  font-size: 30px !important;
  letter-spacing: -0.5px !important;
  line-height: 1.2 !important;
}
```

These `!important` declarations exist to win the cascade against the desktop rules in `main.css`. The root cause is that `main.css` is loaded **after** `responsive.css`'s author intended:

Actual `base.njk:19-23` load order:
```
tokens.css -> reset.css -> main.css -> nav.css -> responsive.css
```

Plan §5 specified:
```
tokens.css -> reset.css -> layout.css -> nav.css -> main.css -> home.css -> services.css -> ...
```

Because `responsive.css` is loaded after `main.css`, its media-query rules should naturally beat `main.css` in the cascade for the same specificity... but `main.css` rules often use compound selectors (`.statement .row`, `.process .head h2`) that out-specify the chained class selectors in `responsive.css`. `!important` is the workaround.

The fix is structural (match specificity by writing `responsive.css` against the same selectors), not just removing the `!important`. Practically: either accept `!important` here, or do the per-section CSS split the plan §5 outlined so each section's mobile rules live next to its desktop rules and the cascade resolves naturally.

### Finding 8 — Inline language switcher in utility-bar bypasses the `language-switcher.njk` partial
**Severity:** Low
**Files:** `src/_includes/partials/utility-bar.njk:12-15`, `src/_includes/partials/language-switcher.njk` (unused)
**Directive:** FRAMEWORK-I18N.md §"Language switcher partial" — `language-switcher.njk` is the canonical mechanism. Plan §3 — "Utility bar (top, dark): phone, email, language switcher (rendered via `partials/language-switcher.njk` using `locale_links`)".

The plan and i18n spec describe a `language-switcher.njk` partial that uses `locale_links` and renders `<nav class="lang-switch">...</nav>`. The partial exists at `src/_includes/partials/language-switcher.njk` but is **not included anywhere** (verified: `grep -rn "language-switcher" src/` returns nothing). Instead, `utility-bar.njk` inlines a two-link switcher using `alternateUrl(lang)`.

Functionally the inline version works, but:
- `locale_links` is the i18n-spec-canonical mechanism, with the documented property that it "only returns languages where the current page actually exists in the other locale." The inline `alternateUrl(lang)` assumes the alternate exists — for paginated pages or pages that exist in only one locale, this would silently produce broken links. (Today every page exists in both locales by design, so this is a latent risk, not a current bug.)
- Two parallel implementations of the same concept means a future change has to update both.

Fix: include `language-switcher.njk` from `utility-bar.njk` instead of inlining, and have `language-switcher.njk` use `locale_links` per spec. Style overrides for the utility-bar position can live in CSS.

**Files:** `src/content/is/index.njk:66, 83, 100, 117, 242`; `src/content/is/thjonusta/index.njk:74`; `src/content/is/about/index.njk:99`; mirror cases in EN tree

### Finding 12 — Hardcoded weight/style in SVG logo references Inter at weights/italics that are not loaded
**Severity:** Low
**Files:** `src/_includes/partials/header.njk:5-6` (and the duplicate in `footer.njk` via `<use>`)
**Directive:** FRAMEWORK-PORT-PROMPT.md §"Conventions" — "No inline styles" (SVG presentation attributes aren't strictly inline CSS, but they're the same pattern). FRAMEWORK-PORT-PROMPT.md §"Web fonts" — self-host the faces the design uses.

The SVG logo wordmark uses:
- `<text font-weight="800">BRUNA</text>` — **weight 800 is not loaded.** `tokens.css` only ships Inter 400 and 600.
- `<text font-weight="600" font-style="italic">þéttingar</text>` — italic 600 is not loaded; only normal 600.

Even after Finding 1 is fixed, the logo will render with **synthetic bold** (browser fakes weight 800 from the loaded 600 face) and **synthetic italic** (browser slants the 600 face). Synthetic faces always look worse than authored ones, especially for the brand wordmark.

Fix one of:
- Self-host Inter 800 and Inter 600 italic (two more `.woff2` files, two more `@font-face` blocks in `tokens.css`).
- Or, change the SVG to use weights/styles the design has actually committed to shipping (e.g., weight 700 instead of 800; rebuild the wordmark without italic).
- Or, accept synthetic rendering as a known compromise and document it.

This is the only place in the codebase where weights/styles outside 400/600-non-italic are requested. The brief's hypothesis #2 (logo SVG inline font-family resolving differently) is partly correct — the synthetic-bold/italic rendering will look slightly off on every page, but the same on both locales, so it isn't the IS-vs-EN cause.

- hreflang chain is emitted on every page (`base.njk:12-14`), including `x-default` -> Icelandic.

## Verdict

**Critical findings: 1.** The font 404 is the root cause of the user's reported typeface symptom and is a one-line fix.

**High findings: 4.** Bypassed `<picture>` pipeline, raw values throughout `main.css`, parallel `:root` definitions with overlapping token names, and the data-file-vs-markdown-collection architectural divergence from plan §7. The first two are explicit framework rule violations; the third is internal inconsistency; the fourth is plan deviation.

**Medium findings: 2.** Missing `Organization` / `AboutPage` JSON-LD, and `!important` use in `responsive.css`.

**Low findings: 3.** Unused language-switcher partial, untranslated "EN"/"IS" labels, hardcoded locale paths in singleton anchors.

**Info: 3.** Directory-data file format choice, SVG logo weight/italic mismatch with loaded faces, `meta.url` pathPrefix coupling.

Total: 13 findings, 1 Critical, 4 High, 2 Medium, 3 Low, 3 Info.

The typeface symptom is fixable in one edit. The rest are spec divergences worth addressing in a follow-up token-and-image migration pass.
